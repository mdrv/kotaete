# v0.0.5

## Summary

This release migrates season score persistence from a local JSON file (`~/.kotaete/state/season-points.json`) to **SurrealDB**, and adds a comprehensive **quiz event logging** system that records all quiz state transitions in real-time. The event log powers the upcoming web spectator frontend (Phase 2).

## Breaking Changes

### SurrealDB is now required

The daemon and CLI commands (`season start/stop/scores/reset`, `quiz run`) now require a running SurrealDB instance. The connection defaults to `http://localhost:596/rpc` with namespace `medrivia` and database `nipbang_kotaete`.

Configure via `SeasonStoreOptions` in code or ensure SurrealDB is running with:

```bash
surreal start --bind 0.0.0.0:596 --user ua --pass japan8 memory
```

### Manual migration required

Existing season scores in `~/.kotaete/state/season-points.json` are **not** automatically migrated. You must manually insert historical scores into SurrealDB using SurrealQL. Example:

```surql
CREATE season SET season_id = 'my-season', group_id = '120363xxx@g.us', status = 'active';
CREATE season_score SET season_id = 'my-season', mid = 'pb25vanya', points = 44, reached_at = 1776758322298, nickname = '', kananame = '', classgroup = '';
```

The old `season-points.json` file is no longer read but is preserved on disk.

## What was implemented

### Phase 1A: Season ID in config

- Added `id` field to `SeasonConfig` in `types.ts` for explicit season identification.
- Added `normalizeSeason()` helper to derive season ID from config (falls back to `groupId` for backward compat).
- Updated `daemon/protocol.ts` relay schema to include `seasonId`.

### Phase 1B: SurrealDB schema for season scoring

- Defined `season` and `season_score` tables with SCHEMAFULL enforcement.
- Fields: `season_id`, `group_id`, `caption`, `status`, `created_at`, `updated_at` on `season`.
- Fields: `season_id`, `mid`, `points`, `reached_at`, `nickname`, `kananame`, `classgroup` on `season_score`.
- Unique indexes on `season_id` and `(season_id, mid)` composite.

### Phase 1C: SeasonStore migration to SurrealDB

- Complete rewrite of `SeasonStore` (`src/quiz/season-store.ts`) to use SurrealDB as the backend.
- Old JSON-file code replaced; constructor still accepts `string` for backward compat (ignored).
- All methods (`addPoints`, `resetGroup`, `setGroupMembers`, `adjustPoints`, `setPoints`, etc.) now execute SurrealQL queries.
- `chain()` pattern ensures serialized writes for consistency.
- `InMemorySeasonStore` preserved for tests (209 tests unchanged).
- Async variants (`getPointsAsync`, `getMembersAsync`, `getReachedAtAsync`) are the primary API; sync methods return empty data.

### Phase 1D: Quiz event logging

- New module `src/quiz/event-logger.ts` ā€” `QuizEventLogger` class.
- Connects to SurrealDB and ensures schema for 4 tables:
  - `quiz_session` ā€” mutable quiz session state (current question, round, accepting answers, deadline).
  - `quiz_event` ā€” append-only event log (question_asked, answer_correct, answer_wrong, timeout, warning, cooldown, god_stage_announced, god_stage_asked, quiz_finished, round_break, special_duplicate).
  - `live_score` ā€” per-member score projection for fast leaderboard queries.
  - `live_member_state` ā€” per-member transient state (cooldown deadline, wrong chances remaining).
- All tables use SCHEMAFULL with typed fields and appropriate indexes.
- Event logging is **fully optional** ā€” if SurrealDB is unavailable, the quiz continues without logging.
- 12 logging call sites integrated into `QuizEngine`:
  1. `moveToNextQuestion()` ā€” question_asked / god_stage_asked
  2. `handleCorrect()` ā€” answer_correct + live_score upsert + member state update
  3. `handleWrong()` (normal) ā€” answer_wrong + live_score + member state
  4. `handleWrong()` (special) ā€” special_duplicate
  5. `handleTimeout()` ā€” timeout
  6. `handleQuestionWarning()` ā€” warning
  7. `finishQuiz()` ā€” quiz_finished + finishSession
  8. `stopCurrentQuiz()` / `stopCurrentQuizWithFinal()` ā€” finishSession('stopped')
  9. `onIncomingMessage()` cooldown block ā€” cooldown
  10. God stage announcement ā€” god_stage_announced
  11. Round break ā€” round_break
- Wired into `DaemonRuntime` with dynamic import (non-fatal on failure).

### Daemon wiring

- `runtime.ts` initializes both `SeasonStore` and `QuizEventLogger` in `start()` with try/catch (non-fatal).
- Both are passed to `QuizEngine` via conditional spread.
- Event logger uses `chain()` for serialized fire-and-forget writes.
