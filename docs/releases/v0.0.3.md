# v0.0.3

## Summary

This release fixes the Baileys "Couldn't Link Device" failure with deterministic postinstall patching, adds an Astro-powered documentation site, and sets up automatic GitHub Pages deployment with both rendered and raw markdown available. It also introduces package rename and CLI tooling for LID↔PN conversion.

## What was implemented

### Follow-up reliability fixes (post-release triage)

- Improved Baileys LID→PN feed reliability:
  - Persist mappings from both `lid-mapping.update` events and `messages.upsert` participant/alt-JID pairs.
  - Added debug logs only when mappings actually change, so new ingestion is observable.
- Improved Baileys run scheduling visibility:
  - `kotaete run` scheduling response now includes intro/start WIB timestamps and remaining delay, so “scheduled but not sending yet” is explicit.
  - Added `--no-schedule` mode so `kotaete run ... --no-schedule` starts quiz intro/questions immediately.
  - Quiz schedule source now prefers `kotaete.ts` inside quiz directory (via `defineConfig({ intro, start })`) instead of relying only on directory naming.
- Improved ARM64 browser runtime support for `wwebjs`:
  - Added executable resolver priority:
    1. `KOTAETE_PUPPETEER_EXECUTABLE_PATH`
    2. `KOTAETE_PLAYWRIGHT_EXECUTABLE_PATH`
    3. Auto-detect latest Playwright Chromium under `~/.cache/ms-playwright/chromium-*`
  - Enables ARM64 environments to use Playwright-installed Chromium without Puppeteer-managed Chrome.

### Package and CLI updates

- Package identity renamed from `nipbang-kotaete` to `@mdrv/kotaete`.
- Quiz schedule config import now requires:
  - `import { defineConfig } from '@mdrv/kotaete'`
- Added new conversion tools:
  - `kotaete tool to-pn <lid>`
  - `kotaete tool to-lid <pn-or-jid>`
  - These resolve values from the persisted local map at `~/.kotaete/state/lid-pn-map.json`.
  - When local mapping is missing, tools now ask the running daemon to perform direct provider lookup against WhatsApp and persist the result back to cache.

### Baileys linking fix

- Pinned `@whiskeysockets/baileys` to `7.0.0-rc.9` (GitHub ref `d077902`).
- Added `scripts/patch-baileys-linking.ts` — idempotent postinstall patcher for:
  - `validate-connection.js`: `passive: true → false`, `lidDbMigrated: false → true`
  - `socket.js`: `await noise.finishInit() → noise.finishInit()`
- Added startup guardrail: `baileys:patch:check` runs before daemon start.
- Updated `src/whatsapp/client.ts` socket config: `browser: Browsers.macOS('Desktop')`, `syncFullHistory: true`.
- Added `docs/baileys-linking-fix-guide.md` with full troubleshooting matrix.

### Documentation site

- Added `docs/astro/` — minimal Astro project using content collections with glob loader.
- Renders all markdown from root `/docs` into navigable HTML pages.
- Index page with guide and release notes listings.
- Dynamic `[...slug].astro` route for each doc entry.
- Handles files with and without frontmatter (fallback title from filename).
- Raw markdown files served at `/markdown/` in deployed output.

### GitHub Pages deployment

- Added `.github/workflows/deploy-docs.yml`:
  - Triggers on push to `main` and `workflow_dispatch`.
  - Builds Astro docs site, copies raw `/docs` into `_docs-dist/markdown`.
  - Deploys using `actions/upload-pages-artifact@v4` and `actions/deploy-pages@v5`.
- Added `docs:build` and `docs:dev` scripts to root `package.json`.

### Release notes

- Added `docs/releases/v0.0.3.md`.
